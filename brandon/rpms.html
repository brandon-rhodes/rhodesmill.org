RPMs



Several programs that are favorites of mine or that I use frequenly for my work are not supported as part of Red Hat Software's offical distributions, so I have packaged them myself and contributed them to Red Hat's contrib/ directory. For an independent listing of my contribution, see my entry on the excellent RPM Repository.
Applications/Networking: Argus 1.7 (beta 1b)

hurricane/readmes/argus-1.7beta1b-1.readme

hurricane/SRPMS/argus-1.7beta1b-1.src.rpm
hurricane/i386/argus-1.7beta1b-1.i386.rpm
hurricane/i386/argus-utilities-1.7beta1b-1.i386.rpm
This is an IP transaction auditing tool that I use to monitor accesses to the machines I administer. I have split it into two components: the sniffer itself (which you will probably need installed on only one machine per subnet), and the collection of commands it comes with for scanning its log files (which you might want to install on many machines).

Applications/Scientific: XEphem 3.0

hurricane/SRPMS/xephem-3.1-1.src.rpm
hurricane/i386/xephem-3.1-1.i386.rpm
noarch/xephem-ppm-1990-2.noarch.rpm
XEphem should be used by every astronomer. Many amateur astronomers spend money for PC-based systems, while this free application outperforms almost all of them! (Exception: it will not do gravitational dynamics simulations, but they are almost irrelevant to the needs of an observer with access to up-to-date ephemeri.) I think every Red Hat user should try this package out.

The ppm star database is included in a separate RPM: it is an optional and very large star catalog that can be used by XEphem.

hurricane/readmes/python-1.5-1.readme

i386/tkinter-1.5-1.i386.rpm
I am a fan and regular user of Python. This is the most recent version, newer than the 1.4 currently being shipped with Red Hat 5.0. When making these packages I attempted to provide a more complete range of functionality than does Red Hat, while carefully retaining Red Hat's package structure. In particular I compiled all of the libraries that could possibly be useful but that did not require other software; these are in the `extralibs' package.

Most sigificantly, I compiled the Python interpreter with threads enabled, taking advantage of the C library and kernel modifications available with Red Hat 5.0. This incurs a performance penalty for nonthreaded applications since the interpreter performs all of its operations in a thread-safe manner, but this will probably not be noticable since Python 1.5 itself is notably faster than 1.4 was.

Development/Tools: CWeb 3.4g

hurricane/readmes/cweb-3.4g-1.readme

hurricane/SRPMS/cweb-3.4g-1.src.rpm
hurricane/i386/cweb-3.4g-1.i386.rpm
Sometime around the seventies, the eminent computer scientist Donald Knuth became dissatisfied with the manner in which computer programming languages force us to organize our code. Actually the problem was more specific: he was using Pascal, which forces your application, however large, to be expressed as a single program module. A subroutine is coded inside the routine that calls it, right smack in between its parent's variable declarations and code! So in a large program there could be thousands of lines separating the variables of a high-level procedure with its code. We are lucky that such an unfortunate language did not drive Dr. Knuth out of computing altogether.

What Knuth did instead was to develop the doctrine of ``literate programming,'' along with an implementation for Pascal that was called Web. His scheme contains two ingredients: first, you express a program hierarchically, breaking it into concentric levels of detail that allow you to view high level structures without having them interrupted by their lengthy components; second, you write a description of what each of these pieces of code does. This structure can then be collapsed into compilable code, or be formatted and printed. The Web system was used by Knuth to create the current version of the widely used TeX program.

CWeb is another implementation of the literate programming concept, written by Silvio Levy and Dr. Knuth. I would estimate that literate programming is probably only half as useful for writing C as it was for Pascal: since C compilers let you combine multiple source files and make you write all functions in the external scope, they avoid the horrible mess that Pascal made of a large project.

But even though C is more convenient, cweb still seems like a good idea. Putting my verbal commentary on a piece of C code into a nearby comment - a fragment of text stuck inside of slashes and stars - seems somewhat lopsided to me. So I am very attracted to the idea of a system that places code and description on an equal footing; thus I have packaged CWeb and plan to use it the next time I write C code.

Development/Tools: Swig 1.1p5

hurricane/swig-1.1p5-1.src.rpm
hurricane/swig-1.1p5-1.i386.rpm
The Simplified Wrapper and Interface Generator is a magnificent tool that automatically builds interfaces to collections of C or C++ functions from any one of several scripting languages. It currently supports Python, Tcl, Perl, and Guile.

The individual who made the RPM for Swig 1.1p3 incorrectly named the packages with the version number `1.3p1'. So this package in fact represents the most recent version.

Games: Adventure

SRPMS/adventure-2.5-1.src.rpm

This is the most recent authorized edition of the venerable and original text adventure game. I recommend playing it late at night in a dark room, as this enhances its atmosphere considerably; if you play it in a lighted room or office you are likely to forget that caves are dark and imperfectly lit by your latern.
